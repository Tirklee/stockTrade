# 股票交易管理系统 - 开发规范

## 项目重大变化规则

### 重大变化的定义

以下情况被视为项目重大变化，必须及时更新说明文件：

1. **目录结构变化**
   - 新增、删除或重命名目录
   - 文件移动到其他位置
   - 目录层级调整

2. **文件变化**
   - 删除不再使用的文件
   - 新增核心功能文件
   - 文件重命名或位置调整

3. **依赖变化**
   - 新增或删除Python依赖包
   - 依赖包版本重大更新

4. **配置变化**
   - 新增配置文件
   - 配置项修改影响启动方式

5. **API变化**
   - 新增、修改或删除API端点
   - 路由路径变化

6. **数据库变化**
   - 表结构修改
   - 新增表或删除表

## 必须更新的说明文件

根据重大变化的类型，更新相应的文档：

| 变化类型 | 必须更新的文档 |
|---------|--------------|
| 目录/文件变化 | `PROJECT_STRUCTURE.md` |
| 新增功能 | `PROJECT_SUMMARY.md` |
| Bug修复 | `BUGFIX_SUMMARY.md` |
| 开发规范变更 | `DEVELOPMENT_GUIDELINES.md` |
| 启动方式变化 | `QUICK_GUIDE.md` |
| 任何重大变化 | `task_progress.txt` |

## 更新要求

1. **及时性**：在完成重大变化后的24小时内更新相关文档
2. **完整性**：确保文档反映最新状态
3. **准确性**：更新内容必须与实际代码一致
4. **可追溯性**：在文档中记录变更日期和变更内容

## 文档维护责任

- **开发者**：负责在提交代码时同步更新相关文档
- **代码审查者**：检查代码变更是否需要更新文档
- **项目负责人**：定期审查文档更新情况

## 版本控制

使用Git进行版本控制，确保：
- 每次重大变化都有清晰的commit message
- 相关文档更新与代码变更一起提交
- 使用标签标记重要版本发布

## 示例

### 好的commit message
```
feat: 添加持仓盈亏计算功能

- 新增 app/services/trade_service.py 中的 calculate_position_pnl 方法
- 更新 PROJECT_STRUCTURE.md 添加新方法说明
- 更新 task_progress.txt 标记完成的任务
```

### 需要更新文档的场景

1. 添加了新文件 → 更新 `PROJECT_STRUCTURE.md`
2. 删除了废弃文件 → 更新 `PROJECT_STRUCTURE.md`
3. 修复了重要bug → 更新 `BUGFIX_SUMMARY.md`
4. 改了启动命令 → 更新 `QUICK_GUIDE.md`

## 禁止事项

1. ❌ 提交代码但不更新相关文档
2. ❌ 文档内容与实际代码不一致
3. ❌ 批量更新多个不相关的功能但只写一条commit
4. ❌ 忽略文档更新要求

## 文档评审

每次代码审查（Code Review）时，应检查：
- [ ] 是否存在重大变化需要更新文档
- [ ] 更新的文档内容是否准确
- [ ] 所有相关的文档是否都已更新

## 联系与反馈

如有文档相关问题或建议，请通过以下方式反馈：
- 提交Issue
- 在代码审查中提出
- 联系项目维护者
